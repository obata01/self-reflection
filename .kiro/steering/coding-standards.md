# コーディング規約

## Docstring

- 全てのクラスとパブリックメソッドに Google Style の Docstring を記述する.
- Docstringは日本語で記載する.
- 説明は簡潔にしてドットで終えること.
- Docstring側へのデータ型の記載は不要（コード側の型ヒントで十分）.
- 必要に応じて Raises セクションも追記する.

```python
def sample_function(arg1: int, arg2: str) -> bool:
    """説明は簡潔にしてドットで終えること.

    Args:
        arg1: 引数1
        arg2: 引数2

    Returns:
        戻り値

    Note:
        - 必要に応じてNoteに補足を記載.
    """
    return True
```

## コメントポリシー

- テストコードを除くソースコードでは、コメントの使用を最小限にする.
- 変数名・関数名・docstring・コードの構造で意図を伝えることを優先する.
- それでも読み手に伝わりにくい箇所にのみコメントを残す.
- セクション区切りとして視認性を高める目的のコメントは許可する.
- `# noqa` 等のリント制御コメントはこのルールの対象外.
- コード側に不要なコメントアウトを入れない.

## テストコード方針

- テストフレームワークは pytest を使用する.
- 極力 `pytest.mark.parametrize` でテストケースを共通化し、テストコード自体の保守性を高めること.
- テストコードは `tests/` ディレクトリに配置する（`src/` 内には置かない）.
- テストファイル名は `test_<対象モジュール名>.py` とする.
